陶辉 : NGINX支持QUIC/HTTP3的实现路径和实践思考

  • HTTP1

    • 请求
      • start-line(request-line / status-line)
      • header-field
      • message-body
    • 语义
      • (Client Request / Server Response机制)
      • Header + body编码协议
    • 问题
      • 无状态（会放大编码效率低的问题）
      • 编码效率低
      • 慢启动（慢慢打起来流量，带宽打起来比较慢）
      • 建连接（最后一公里问题）
      • 异步编程开发效率（没懂为什么同步比异步更好）
      • 长连接中止请求代价高（终止连接需要重新建）
      • 不支持服务器推送(接受服务器消息)
      • HTTP不支持流控

  • HTTP2

    • 特性
      • 多路复用
        • Stream级流控
        • 带权重优先级
        • Reset置位（可以复用一个连接）
      • HPACK头部编码（动态表）
      • 服务器消息推送
    • 问题
      • 队头阻塞（队头出问题影响后面的frame）
      • 建连接（两次握手：传输层和TLS）

  • HTTP3

    • 标准

  • 连接迁移
    • 固定的连接ID
    • 连接变了：PATH_CHALLENGE Frame
      • PMTU验证
      • 拥塞控制清零
  • 概念
    • Packet
      • Long Packet Header/Short Packet Header
    • QUIC Frame
      • 不可跨越Packet
      • Stream有序字节流
        • 双向Stream
        • 单向Stream
          • QPACK动态表、服务器推送
      • Packet 源连接ID / 目的连接ID
      • 如何避免恶意FLOOD（Retry）
    • HTTP3 Frame
      • 可跨越多个Packet
    • Message
